# ADAN Library Development Plan

This file describes the next library work that should happen in `libs/`, in order.

The plan follows the patterns already used by the current standard libraries:

- one top-level aggregator file per library, such as `string/string.adn` and `process/process.adn`
- several small focused `.adn` files per library
- a C backend only when ADAN cannot realistically implement the feature on its own

## Planning rules used here

1. Prefer pure ADAN when the feature is only string/array/math logic.
2. Use a C backend for OS, filesystem, socket, timer, thread, or window APIs.
3. Reuse existing libraries when possible:
	 - `adan/string`
	 - `adan/io`
	 - `adan/process`
	 - `adan/math`
	 - `adan/json`
4. Avoid callback-heavy APIs for now. The current grammar does not expose function types or closures well enough for an ergonomic callback-based UI API. Prefer polling and record returns.
5. Keep names simple and flat, matching the current style.

---

## Recommended build order

1. `adan/path`
2. `adan/fs`
3. `adan/time`
4. `adan/test`
5. `adan/os`
6. `adan/window`
7. `adan/webview`
8. `adan/thread`
9. `adan/net`
10. `adan/http`
11. `adan/ui` (defer until much later)

Why this order:

- `path`, `fs`, `time`, and `test` immediately improve day-to-day programming.
- `window` is the real foundation for native UI work.
- `webview` is the fastest route to practical desktop UI.
- `thread`, `net`, and `http` are useful, but they are not prerequisites for the first UI story.
- `ui` should not be attempted until the lower layers and language ergonomics are stronger.
